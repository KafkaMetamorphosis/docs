# Kafka Cluster Management

This feature provides CRUD operations for managing Kafka cluster registrations. A cluster registration tells Franz where a Kafka broker lives so that the reconciler can create, update, and delete topics on it.

The cluster register servers only as a discovery to the clusters to be used by the topics, acls and so that franz take care of. Franz does not manage/interact with the clusters like brokers and infrastructure resource.

## Concepts

| Concept | Description |
|---|---|
| **Cluster** | A registered Kafka cluster that Franz manages. Each cluster has a unique name, a bootstrap URL for broker connectivity, and optional labels for organization and filtering. The cluster record is the target against which topic claims are reconciled. |
| **Labels** | Can be used as a metadata store to add any information to the cluster and in logs to find, associate and match the cluster with other resources. |

## Data Model

### Cluster

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | `string` | Yes | Unique identifier for the cluster. Immutable after creation. Used as the natural key in API paths. |
| `default-topic-configuration-id` | `uuid` | Yes | Is the default topic configuration for the given cluster. |
| `bootstrap-url` | `string` | Yes | Kafka bootstrap server URL (e.g. `broker-1:9092,broker-2:9092`). |
| `labels` | `map<string, string>` | No | Arbitrary key-value metadata for categorization and filtering. Defaults to an empty map. |

## HTTP Endpoints

All endpoints are served under `/api/v0/`.

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/v0/clusters` | List all clusters, with pagination. |
| `POST` | `/api/v0/clusters` | Create a new cluster. |
| `GET` | `/api/v0/clusters/:cluster-name` | Return the details of a specific cluster. |
| `PUT` | `/api/v0/clusters/:cluster-name` | Update a specific cluster. |
| `DELETE` | `/api/v0/clusters/:cluster-name` | Delete a specific cluster data. |

## Pagination

List endpoints accept `page` and `size` query parameters.

| Parameter | Type | Default | Description |
|---|---|---|---|
| `page` | `integer` | `1` | Page number (1-indexed). |
| `size` | `integer` | `20` | Number of items per page. |

Example: `GET /api/v0/clusters?page=2&size=10`

# Management Features

## Taint

Taints are defined in the cluster labels; tolerations are defined in the topic-definition labels. A topic claim is only placed on a cluster when that cluster's taints are either absent or tolerated by the topic definition.

### Effects

| Effect | Description |
|---|---|
| `no-creation` | Blocks new topic claims from being placed on this cluster. Existing claims are unaffected. |
| `drain` | Actively migrates all topic claims off this cluster to other eligible clusters. Overrides tolerations — no topic can remain. |

### Syntax

Taint on a cluster label:
```json
{
    "franz/taint/<taint-name>": "<effect>"
}
```

Toleration on a topic-definition label (to allow placement on a `no-creation` tainted cluster):
```json
{
    "franz/toleration": "<taint-name>:no-creation"
}
```

### Examples

**Block new topics unless explicitly tolerated:**
```json
// cluster labels
{ "franz/taint/restricted": "no-creation" }

// topic-definition labels (opt-in to this cluster)
{ "franz/toleration": "restricted:no-creation" }
```

**Drain all topics from a cluster:**
```json
// cluster labels
{ "franz/taint/decommission": "drain" }
```
Once this label is set, Franz will relocate all claims on this cluster to other eligible clusters.

### When to use taints

| Scenario | Taint to apply | Why |
|---|---|---|
| **Cluster decommissioning / migration** | `decommission:drain` | Progressively move all topic claims to newer clusters without manual intervention. |
| **Incident isolation** | `unstable:no-creation` | Stop new topics landing on a degraded cluster while keeping existing ones running. |
| **Dedicated / reserved clusters** | `dedicated-<team>:no-creation` | Enforce deny-by-default access; only topics with a matching toleration can be placed here. |
| **Proactive capacity management** | `capacity:no-creation` | Pause new placements before hitting governance hard limits (e.g. `avg-replica-per-broker > 3000`). |
| **Maintenance windows** | `maintenance:no-creation` | Prevent new claims from entering `Applying` state during a rolling broker restart. |

### Design notes

- A cluster may carry multiple taints using namespaced keys: `franz/taint/foo`, `franz/taint/bar`.
- `drain` always wins — it cannot be tolerated. `all:drain` is a shorthand that applies to every topic on the cluster.
- During a drain, the safe migration order is: create new claim on target cluster → wait for `Synced` → delete old claim. This means a topic temporarily exists on two clusters.
- If no eligible cluster exists for a drained topic, the claim remains and Franz retries until one becomes available.

## Affinities

Affinities express a preference or requirement for clusters with a specific label. A topic definition with an affinity will only be placed on clusters that carry the matching label.

Cluster label:
```json
{
    "my-fleet/tier": "high-volume"
}
```

Topic-definition label:
```json
{
    "franz/affinity": "my-fleet/tier=high-volume"
}
```

**When to use:** Route topics that require high throughput, low latency, or specific hardware tiers to the right cluster pool.

## Anti-Affinities

Anti-affinities exclude clusters that carry a specific label. A topic definition with an anti-affinity will never be placed on a cluster that matches.

Cluster label:
```json
{
    "my-fleet/tier": "high-volume"
}
```

Topic-definition label:
```json
{
    "franz/anti-affinity": "my-fleet/tier=high-volume"
}
```

**When to use:** Prevent cost-sensitive or low-priority topics from consuming capacity on premium cluster tiers.

## Scheduling Pipeline

When placing a topic claim, Franz evaluates clusters in this order. A cluster must pass every stage to be eligible:

1. Exclude clusters with `drain` effect (nothing schedules here)
2. Exclude clusters with `no-creation` taints the topic definition does not tolerate
3. Exclude clusters that match the topic definition's anti-affinities
4. Exclude clusters that do not match the topic definition's affinities (if any are set)
5. Apply cluster-level governance checks (e.g. replica/leader limits)
6. Select from remaining candidates (strategy TBD: round-robin, least-loaded, etc.)
