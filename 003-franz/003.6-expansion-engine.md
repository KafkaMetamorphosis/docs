# Topic Definition Expansion Engine

The expansion engine is the core scheduling system in Franz. It determines which Kafka clusters should host a given topic by evaluating taints, tolerations, affinity selectors, weights, and shard-size constraints. The engine then creates or updates TopicClaims and TopicRevisions accordingly.

## Overview

When a TopicDefinition is created, updated, or deleted, Franz must decide which clusters the topic should exist on and ensure the correct TopicClaims and TopicRevisions are in place. This process is called **expansion**.

The expansion engine is implemented as pure logic (`franz.logic.expansion`) orchestrated by a transactional operations layer (`franz.ops.expansion`).

---

## Trigger Points

| Event | Expansion Behaviour |
|---|---|
| TopicDefinition **created** | Expand inline (synchronous) during the create request. |
| TopicDefinition **updated** (labels or config change) | Re-expand inline during the update request. |
| TopicDefinition **deleted** (soft delete) | Create `PendingDelete` revisions for all active claims. |
| Manual trigger endpoint | Expands all definitions with `expansion_status = PendingExpansion`. |

## Manual Expansion Endpoint

| Method | Path | Description |
|---|---|---|
| `POST` | `/api/v0/expand_topics_claims` | Expand topic definitions given the topics that matches. |

### Request

```json
{
    "topics": ["topic-list or regex"],
    "dry-run": false

```

Pass `"topics": ["*"]` to expand all definitions with `expansion_status = PendingExpansion`.

## Scheduling Pipeline

The expansion pipeline for a single TopicDefinition proceeds through these stages:

### 1. Drain Separation

All clusters are partitioned into two groups:
- **Draining clusters**: clusters with a `franz.taint` label whose effect is `drain`. These are processed separately to create `PendingDelete` revisions for any existing active claims.
- **Non-draining clusters**: all other clusters, which proceed to the taint filter.

### 2. Taint Filter

For each non-draining cluster, evaluate whether the cluster's taint allows placement:

| Cluster Taint | Topic Definition Toleration | Result |
|---|---|---|
| No taint | (any) | **Allowed** |
| `<name>:no-creation` | Contains `<name>:no-creation` | **Allowed** |
| `<name>:no-creation` | Does NOT contain `<name>:no-creation` | **Blocked** |
| `<name>:drain` | (any) | **Blocked** (drain always blocks) |

Taint syntax in cluster labels:
```json
{ "franz.taint": "restricted:no-creation" }
```

Toleration syntax in topic definition labels (comma-separated for multiple tolerations):
```json
{ "franz.taint/toleration": "restricted:no-creation,maintenance:no-creation" }
```

### 3. Affinity Selector Filter

If the topic definition has an `franz.affinity/selector` label, only clusters whose labels match ALL key=value pairs in the selector pass this stage.

Selector syntax (comma-separated):
```json
{ "franz.affinity/selector": "my-fleet/tier=high-volume,my-fleet/env=prod,my-fleet/geo=br" }
```

If no selector is present, all taint-passing clusters are eligible.

### 4. Weight-Based Ranking

Eligible clusters are sorted by descending `franz.affinity/weight` label value. Clusters without a weight default to `0`.

```json
{ "franz.affinity/weight": "3" }
```

### 5. Shard-Size Limit

Shard-size controls how many clusters a topic should be placed on. It is specified as a label on the topic definition:

```json
{ "franz.affinity/shard-size": "2" }
```

**Shard-size only applies when affinity selectors are present.** If no selectors are defined, all taint-passing clusters are eligible regardless of shard-size.

When shard-size is set:
- Take the top N clusters by weight (where N = shard-size)
- If fewer than N clusters are eligible, the expansion status remains `PendingExpansion`

When shard-size is not set but selectors are present:
- All matching clusters are selected (no limit)

### Pipeline Summary

```
All clusters
  |
  +--> [Drain separation] --> draining clusters --> PendingDelete for active claims
  |
  +--> Non-draining clusters
         |
         +--> [Taint filter] --> taint-passing clusters
                |
                +--> [Selector filter] (if selectors present) --> matching clusters
                |      |
                |      +--> [Weight sort] --> sorted clusters
                |             |
                |             +--> [Shard-size limit] (if shard-size set) --> final selection
                |
                +--> (no selectors) --> all taint-passing clusters = final selection
```

## Configuration Materialisation

When a TopicRevision is created, the engine materialises the topic configuration using a three-layer merge:

```
Layer 1 (base):     Cluster default topic configuration
Layer 2 (override): TopicDefinition topic configuration
Layer 3 (override): TopicClaim configuration override (if present)
```

Each higher layer overrides values from the layer below. The merged result includes:

| Field | Source |
|---|---|
| `partitions` | Most specific layer wins |
| `replication-factor` | Most specific layer wins |
| `retention-ms` | Most specific layer wins |
| `configs` | Deep-merged across all layers (higher layer values override) |

The materialised configuration is stored as the `topic_configuration` field on the TopicRevision. This snapshot captures the exact config at revision creation time.

## Stale-Revision Guard

The expansion engine includes a guard against creating revisions for claims that already have a non-terminal revision in flight.

A revision is **non-terminal** if its status is `PendingReconciliation` or `PendingDelete`. These revisions are still being processed by Gregor Samsa.

When the engine encounters an existing claim whose latest revision is non-terminal:
- It does NOT create a new revision for that claim
- It sets the definition's expansion status to `PendingExpansion` instead of `Expanded`

This ensures Gregor Samsa is not flooded with conflicting revisions for the same claim. The definition will be re-expanded on the next manual trigger or scheduled expansion pass.

---

## Expansion Status

| Status | Meaning |
|---|---|
| `Expanded` | All eligible clusters have claims and all new revisions were created. |
| `PendingExpansion` | Expansion is incomplete. Reasons: stale-revision guard triggered, shard-size not met (fewer eligible clusters than requested), or expansion has not yet been attempted since a cluster change. |

---

## Deletion Expansion

When a TopicDefinition is soft-deleted (status set to `Deleted`), the engine runs a deletion plan:

1. Find all active (non-`Deleted`) claims for the definition
2. For each claim that does not already have a `PendingDelete` revision, create one
3. Set each claim's status to `Deleted`

Gregor Samsa will then pick up the `PendingDelete` revisions, remove the topics from Kafka, and report the `deleted` outcome back to Franz.

---

## Expansion Plan Output

The pure logic function `compute-expansion-plan` returns a plan with these fields:

| Field | Type | Description |
|---|---|---|
| `claims-to-upsert` | `[{:cluster-id, :status, :labels}]` | New claims to create (or reactivate from Deleted). |
| `revisions-to-create` | `[{:cluster-id, :existing-claim-id, :status}]` | Revisions to create for new or existing claims. |
| `claims-to-drain` | `[claim-id]` | Claims on draining clusters to mark Deleted. |
| `drain-revisions` | `[{:claim-id, :cluster-id, :status}]` | PendingDelete revisions for draining claims. |
| `expansion-status` | `string` | `"Expanded"` or `"PendingExpansion"` |

---

## Transaction Semantics

The orchestration layer (`franz.ops.expansion`) executes the full plan inside a single database transaction:

1. Upsert claims for new placements
2. Create `PendingReconciliation` revisions with materialised config
3. Create revisions for existing claims that need updating
4. Create `PendingDelete` revisions and mark drain claims as `Deleted`
5. Update the definition's `expansion_status`

If any step fails, the entire transaction rolls back. This guarantees that claims and revisions are always consistent.

---

## Error Isolation

Each definition is expanded independently. If one definition fails, it does not block the others:

- Failed definitions remain `PendingExpansion`
- Errors are logged and if it blocks claims, the payload of the request should have a `not-expanded` object

## Example: Full Expansion Walkthrough

Given:
- **Cluster A**: labels `{"my-fleet/tier": "high-volume", "my-fleet/env": "prod", "franz.affinity/weight": "3"}`
- **Cluster B**: labels `{"my-fleet/tier": "high-volume", "my-fleet/env": "prod", "franz.affinity/weight": "1"}`
- **Cluster C**: labels `{"my-fleet/tier": "high-volume", "my-fleet/env": "prod", "franz.taint": "restricted:no-creation", "franz.affinity/weight": "2"}`
- **Cluster D**: labels `{"my-fleet/tier": "low-volume", "my-fleet/env": "staging"}`

TopicDefinition `user-events` with labels:
```json
{
    "franz.affinity/selector": "my-fleet/tier=high-volume,my-fleet/env=prod",
    "franz.affinity/shard-size": "2"
}
```

**Pipeline execution:**

1. **Drain separation**: No clusters are draining.
2. **Taint filter**: Cluster C has `restricted:no-creation` but `user-events` has no toleration for it. Cluster C is blocked. Clusters A, B, D pass.
3. **Selector filter**: Selector is `tier=high-volume AND env=prod`. Cluster D (`low-volume`, `staging`) does not match. Clusters A, B pass.
4. **Weight sort**: A (weight=3) comes before B (weight=1).
5. **Shard-size limit**: shard-size=2, so take top 2: A, B.

**Result**: TopicClaims created for Cluster A and Cluster B, each with a `PendingReconciliation` revision. Expansion status: `Expanded`.
