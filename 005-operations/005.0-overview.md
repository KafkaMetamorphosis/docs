# Operations Overview

This document covers deployment, configuration, and operational procedures for the KafkaMetamorphosis system.

---

## System Components

| Component | Role | Instances | Deployment |
|---|---|---|---|
| **Franz** | Control plane. Central API, state store, expansion engine. | 1 (or HA pair behind load balancer) | Standalone JVM process + PostgreSQL |
| **Gregor Samsa** | Reconciler. Polls Franz, applies changes to a single Kafka cluster. | 1 per Kafka cluster | Standalone JVM process, co-located with or near its target cluster |

---

## Deploying Franz

### Prerequisites

- JDK 11+ (tested with JDK 17)
- PostgreSQL 14+ (accessible from the Franz host)
- Leiningen (for building) or a pre-built uberjar

### Building

```bash
cd franz
lein uberjar
```

The uberjar is produced at `target/uberjar/franz-standalone.jar`.

### Running

```bash
java -jar franz-standalone.jar
```

Or during development:

```bash
cd franz
make run
```

The `make run` target waits for the database to be ready (via `wait-for-db`) before starting the application with `lein run`.

### Database Setup

Franz uses Flyway for schema migrations. Migrations are applied automatically on startup from `resources/db/migrations/`.

To start a local PostgreSQL instance:

```bash
make services-up   # starts SERVICES=postgres via Docker Compose
```

### Configuration

Franz uses [aero](https://github.com/juxt/aero) for configuration, read from `resources/config.edn`. All values can be overridden via environment variables.

| Environment Variable | Default | Description |
|---|---|---|
| `PORT` | `8080` | HTTP server listen port |
| `DB_HOST` | `localhost` | PostgreSQL host |
| `DB_PORT` | `5432` | PostgreSQL port |
| `DB_NAME` | `franz` | Database name |
| `DB_USER` | `franz` | Database user |
| `DB_PASSWORD` | `franz` | Database password |

### Component System

Franz uses Stuart Sierra Component for lifecycle management. The system topology is:

```
:config       -- loads config.edn
:database     -- PostgreSQL connection pool (depends on :config)
:api          -- Compojure route handler (depends on :database, :config)
:http-server  -- Jetty HTTP server (depends on :config, :api)
```

### Health Endpoints

| Path | Description |
|---|---|
| `GET /ops/health` | Basic health check |
| `GET /ops/liveness` | Liveness probe (for Kubernetes) |
| `GET /ops/readiness` | Readiness probe (for Kubernetes) |
| `GET /ops/config/dump` | Dump active configuration (exclude in production) |

## Deploying Gregor Samsa

Gregor Samsa is not yet implemented. When available, deployment will follow this pattern:

### One Instance Per Cluster

Each Gregor Samsa instance is configured to manage exactly one Kafka cluster. It identifies itself to Franz by the cluster name.

### Configuration (Expected)

| Environment Variable | Description |
|---|---|
| `FRANZ_URL` | Base URL of the Franz API (e.g., `http://franz:8080`) |
| `CLUSTER_NAME` | Name of the Kafka cluster this instance manages (must match a registered cluster in Franz) |
| `KAFKA_BOOTSTRAP_SERVERS` | Kafka bootstrap URL (for direct Kafka admin operations) |
| `POLL_INTERVAL_MS` | How often to poll Franz for pending revisions (default: 5000) |
| `MAX_RETRIES` | Maximum reconciliation attempts before giving up on a revision (default: 3) |

### Deployment Pattern

```
Kafka Cluster A  <-->  Gregor Samsa A  <-->  Franz API
Kafka Cluster B  <-->  Gregor Samsa B  <-->  Franz API
Kafka Cluster C  <-->  Gregor Samsa C  <-->  Franz API
```

Each Gregor Samsa instance only talks to its assigned cluster and to Franz. There is no communication between Gregor Samsa instances.

## Manual Expansion Trigger

The expansion endpoint is used when cluster changes (create/update) have marked definitions as `PendingExpansion` but expansion has not yet run.

```bash
curl -X POST http://franz:8080/api/v0/topic_definitions_expansion
```

### When to Use

- After creating or updating clusters (since cluster changes do not expand inline)
- After recovering from a failed expansion batch
- As part of a scheduled job to ensure all definitions are fully expanded

### Automation

In production, consider running the expansion trigger on a schedule (e.g., every 5 minutes) to ensure definitions do not remain in `PendingExpansion` indefinitely. This can be a simple cron job or a Kubernetes CronJob.

## Observability

### What to Monitor

| Metric / Check | Why |
|---|---|
| Topic definitions in `PendingExpansion` status | Expansion is incomplete; may need manual trigger or investigation |
| Topic definitions in `Error` status | At least one claim failed reconciliation |
| Topic claims in `Error` status | Reconciliation failure on a specific cluster |
| Topic revisions in `PendingReconciliation` or `PendingDelete` for extended periods | Gregor Samsa may be down or unable to reach the Kafka cluster |
| Revision `attempts` count | High attempt counts indicate persistent reconciliation failures |
| Franz API response times and error rates | Standard API health |
| PostgreSQL connection pool utilisation | Database saturation |
| Gregor Samsa poll success rate | Connectivity between reconciler and control plane |

### Logging

Franz logs structured messages via `clojure.tools.logging`. Key log entries include:

- `topic-definition created/updated/deleted` with topic name
- `expansion complete` with expanded and pending counts
- `expansion failed` with definition ID on errors

### Database Queries for Operational Checks

Count definitions that need expansion:
```sql
SELECT count(*) FROM topic_definitions
WHERE expansion_status = 'PendingExpansion' AND status = 'Active';
```

Find claims in error state:
```sql
SELECT tc.id, td.topic_name, c.name as cluster_name, tc.error
FROM topic_claims tc
JOIN topic_definitions td ON tc.topic_definition_id = td.id
JOIN clusters c ON tc.cluster_id = c.id
WHERE tc.status = 'Error';
```

Find stale pending revisions (older than 10 minutes):
```sql
SELECT tr.id, tr.status, tr.created_at, c.name as cluster_name
FROM topic_revisions tr
JOIN clusters c ON tr.kafka_cluster_id = c.id
WHERE tr.status IN ('PendingReconciliation', 'PendingDelete')
  AND tr.created_at < now() - interval '10 minutes';
```

## Disaster Recovery

### Database

Franz is stateless aside from PostgreSQL. The database is the single source of truth. Regular PostgreSQL backups (pg_dump or continuous WAL archiving) protect against data loss.

### Recovery Procedure

1. Restore the PostgreSQL database from backup.
2. Start Franz. Migrations will verify schema consistency.
3. Run the manual expansion trigger to reconcile any state drift.
4. Restart all Gregor Samsa instances. They will resume polling from where they left off (revisions are persistent).

### Gregor Samsa Failure

If a Gregor Samsa instance crashes or becomes unreachable:
- Pending revisions remain in `PendingReconciliation` or `PendingDelete` in the database.
- No data is lost. When Gregor Samsa restarts, it polls for pending revisions and resumes.
- Topics on the Kafka cluster are unaffected (Kafka is independent of Gregor Samsa uptime).
